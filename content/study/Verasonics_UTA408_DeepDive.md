# (US学习)分析Verasonics UTA-408-GE-MUX接口

> 基于 `VantageNXT-2.3.2-p2/Example_Scripts/Biomedical/VantageNXT_64/UTA-408-GE-MUX/L11-5gH/` 下全部脚本源码逐行分析，并核对官方《Sequence Programming Manual》《Doppler》HelpText 得出。示例探头为 L11-5gH（128 阵元线阵），结论对同目录下 C5-2gH / P4-2gH 同样适用，个别脚本的具体硬编码数值需自行核对。

## 0. 起点：128 个阵元，64 条通道

Vantage NXT 64 通道系统只有 64 路独立收发电子通道，但 L11-5gH 有 128 个物理阵元。要用 64 路硬件去伺候 128 个阵元，只有两条路：

- **写死配对**：每个通道永久接 2 个阵元。简单，但牺牲孔径灵活性。
- **动态开关**：用高压继电器阵列（HVMux）在不同时刻把 64 个通道接到不同的阵元子集上。

`UTA-408-GE-MUX`（Universal Transducer Adapter，型号 408，GE 制式连接器，带 MUX）就是干第二件事的适配板。脚本里的落地代码是这三行：

```matlab
Trans.name = 'L11-5gH';
Trans = computeTrans(Trans);           % 查表得到 L11-5v 同款的几何/频率参数
Trans = computeUTA408GEMux64(Trans);   % 附加 HVMux 开关映射表——后续所有孔径计算都靠它
```

探头命名里的 "gH" 后缀（区别于 "v" 结尾的直连版本）就是"同一批阵元、换 GE 连接器、走 HVMux"的标记。

| 探头 | 阵元数 | 频率 | 阵列类型 |
|---|---|---|---|
| L11-5gH | 128 | 7.6 MHz | 线阵 |
| C5-2gH | 128 | 3.3 MHz | 凸阵 |
| P4-2gH | 96 | 3.15 MHz | 相控阵（虚拟顶点） |

## 1. "合成孔径"：跟文献里的 Synthetic Aperture Imaging 不是一回事

脚本注释里到处写着"2-1"/"3-1 合成孔径"，容易联想到超声文献里经典的 **Synthetic Aperture Imaging（SAI/SAFT）**——用单阵元或小子孔径发射**发散波**照亮全视野，多次发射后把所有收发组合相干合成，换取每个像素点都有独立的发射+接收动态聚焦，代价是单次发射能量分散、需要很多次发射。

这里的"N-1 合成孔径"说的是另一件事：**纯粹是 64 通道数不够，分 N 次连接不同的阵元子集，把数据在空间上拼接起来，凑齐"128 个阵元本该采到的数据"**——不涉及发射方式的改变，也不是为了合成更好的聚焦。Flash 该是平面波还是平面波，128RyLns 该是聚焦逐线还是聚焦逐线，唯一区别是原生 128/256 通道系统不需要"分 N 次"这道工序。更准确的名字应该是"**因通道复用而分批拼接的物理孔径**"。

唯一沾点文献意义"相干合成"的地方，是下文 128RyLns 接收端用 `accumIQ` 把两次子孔径的波束合成结果相干叠加——但那也只是同一发射位置的两次接收合并，跟 SAI 里"多个不同发射位置的数据互相合成"完全不是一回事。（Verasonics SDK 里确实另有专门的发散波/虚拟源合成孔径成像示例，但不在这个文件夹里，是另一条技术路线。）

## 2. 发射端的四种孔径策略

实测下来，L11-5gH 目录下的脚本用了四种截然不同的 MUX 用法：

| 策略 | 代表脚本 | 发射孔径 | 接收孔径 | 每条线/角度采集次数 |
|---|---|---|---|---|
| A：全阵元配对绑定 | `AcquireRF` | 128（配对绑定，1次） | 64（分2次） | 1发射 + 2接收 |
| B：滑动整窗平面波 | `Flash` / `FlashAngles` | 64（3个重叠窗口） | 42/44（窗口内取段） | 3（×na个角度） |
| C1：小孔径+全阵列接收 | `128RyLns` | ≤64（默认32，聚焦） | 128（固定两半+相干合并） | 2 |
| C2：小孔径+窗口内接收 | `WideBeam`系列 | ≤64（默认32，弱聚焦） | 64（=发射窗口） | 1 |

### 策略 A：全阵元配对绑定（`AcquireRF`）

```matlab
TX.Apod = ones(1,Trans.numelements);   % 128 阵元全标记发射
TX.aperture = Paper(1);                % computeMuxAperture(ones(1,128), Trans)
```

当请求"全部 128 阵元同时发射"时，HVMux 会让**每个系统通道同时接 2 个物理阵元**（脚本注释原话："each transmitter connecting to two elements"），物理上真的一次性打满整个阵列。这在平面波发射里可行，因为同一对里两个阵元离得很近，配对造成的相位误差可以忽略。

但**接收端不能这样配对**——配对接收等于把两路回波直接相加，空间信息被销毁，无法再分离。所以接收依然要分 2 次，各自连接真正独立的 64 通道：

```
发射(1次): tttttttttttttttttttttttttttttttttttttttttttttttttttttttttttttttt  (128阵元配对绑定)
接收1:     rrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrr--------------------------------  (阵元1-64)
接收2:     --------------------------------rrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrr  (阵元65-128)
```

### 策略 B：滑动整窗，三次重叠覆盖（`Flash` / `FlashAngles`）

没有配对绑定，每次发射用一个真实独立、1:1 连接的 64 阵元窗口，**窗口内全部阵元都真实发射**。三个窗口重叠 32 个阵元滑过全阵列：

```matlab
Papod = [ ones(1,64) zeros(1,64); ...            % 窗口1：阵元 1-64
         zeros(1,32) ones(1,64) zeros(1,32); ...  % 窗口2：阵元 33-96
         zeros(1,64) ones(1,64) ];                % 窗口3：阵元 65-128
```

接收则反过来，在每个窗口内部只取互不重叠、正好拼满 128 阵元的一段：

```
                窗口1(1-64)发射 ─┐        窗口2(33-96)发射 ─┐       窗口3(65-128)发射 ─┐
接收: 阵元 1–42 ◄────────────────┘  阵元 43–86 ◄─────────────┘  阵元 87–128 ◄──────────┘
```

发射窗口重叠是为了让接收段边界处的阵元不受连接器边缘效应影响；接收孔径 42+44+42=128，无缝拼满。`FlashAngles` 只是在这 3 个窗口基础上再乘 `na` 个转向角度，孔径逻辑不变。

**为什么这里不能像 `AcquireRF` 那样用"配对绑定"一次打满128个阵元？** 答案跟"转向角"直接相关，而不是随便选的实现方式。

官方《Sequence Programming Manual》第14章确认，HVMux 确实支持"一个通道并联接多个阵元"这种工作模式（`Trans.HVMux.type = 'perEL'`：每个阵元一个独立开关，**允许把多个阵元并联到同一个系统通道**），这正是 `AcquireRF` 里 `TX.Apod=ones(1,128)` 触发的机制。手册里一个几乎同构的例子进一步确认了配对的具体方式：

> Since the MUX wiring... allows a channel to be connected to **either element n or n+64**...

结合 `AcquireRF` 脚本本身的证据——它的两次接收窗口正好是"阵元1-64"和"阵元65-128"，用的是同一套底层通道映射——可以确认：`TX.Apod=ones(1,128)` 触发的配对方案是**通道 k 同时接通阵元 k 和阵元 k+64**。对 L11-5gH（阵元间距0.3mm）来说，这两个被绑在一起的阵元实际相距约 **64×0.3mm ≈ 19.2mm**，是阵列近一半宽度的距离，绝不是"挨着的两个阵元"。

这个距离决定了转向角能不能用配对。发射延迟的理想值是位置的函数：`delay(x) = x·sinθ/c`。

- **`AcquireRF`（`Steer=[0,0]`，未转向平面波）**：理想延迟在全部128个阵元上**严格等于0**，跟位置无关。被绑在一起的两个阵元（不管隔多远）本该拿到完全相同的延迟（=0），配对绑定误差为零，物理上无损。
- **`FlashAngles`（`Steer=[angle,0]`，`angle≠0`）**：理想延迟随位置线性变化。被绑在一起的两个阵元相距19.2mm，对任意非零转向角，这两个位置该有的理想延迟会相差一大截（比如15°转向角下，19.2mm的位置差对应的延迟差远超过一个波长）——但一个通道只有一路波形发生器，**物理上只能给这一对阵元共享同一个延迟值**，没法各给各的。所以只要转向角不为零，"配对绑定"这条路在物理上就走不通，脚本只能退回到"3个真正独立、1:1连接的64阵元窗口滑动覆盖全阵列"——每个窗口内64个阵元都是真实独立通道，能各自拿到精确到位置的延迟，代价是要打3次而不是1次。

### 策略 C1：小孔径滑窗 + 全阵列接收（`128RyLns`，经典聚焦成像）

发射端用远小于 64 的聚焦子孔径（默认 `numTx=32`，真实聚焦延迟 `TX.focus=100`波长，落在成像深度内），随 128 条线平移，MUX 只是滑动选一个包含该子孔径的 64 阵元连接窗口。

**接收端不跟着窗口走**，而是固定用两个不重叠的 64 阵元区块，**同一个发射打两次**，分别配这两个固定接收窗口：

```matlab
Receive(k+j).Apod(1:64)    = 1.0;  Receive(k+j).aperture   = Paper(1);   % 固定：阵元1-64
Receive(k+j+1).Apod(65:128)= 1.0;  Receive(k+j+1).aperture = Paper(96);  % 固定：阵元65-128
Event(n).tx = (j+1)/2;  Event(n+1).tx = (j+1)/2;   % 同一个TX打两次，只是配的接收窗口不同
```

以线 64（阵列中央）为例：

```
元素号：       1                    64  65                    128
发射(第1次): tttttttttttttttttttttttttttt     (64阵元窗口，实际驱动中间约32个)
  接收窗口1: rrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrr      (阵元1-64)
发射(第2次): tttttttttttttttttttttttttttt     (相同聚焦延迟，重打一次)
  接收窗口2:                                    rrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrr  (阵元65-128)
             └──────────── 重建时用 accumIQ 相干合并两次接收结果 ────────────┘
```

重建阶段用 `ReconInfo` 的 `accumIQ_replaceIntensity` 模式把两次的波束合成结果做 IQ 域相干累加——数学上等价于"用128个阵元做一次完整动态接收孔径波束合成"，只是分两步算。所以：

- **`_64` 文件名后缀指的是"给64通道系统写的脚本"，不是"接收孔径卡死在64"**。64 是硬件每次采集能同时数字化的阵元上限（`Resource.Parameters.numRcvChannels`），物理限制，改不了脚本。但重建时**实际用于波束合成的有效接收孔径可以达到全部128阵元**（具体多宽取决于深度自适应F值逻辑），并不会被"每次只有64通道"卡死分辨率。
- **真正该调、也能调的分辨率参数是 `P.numTx`**（发射子孔径宽度），脚本自带基于F值的UI滑块，跟"64"这个硬件常数无关。

**这跟教科书"TX/RX孔径共同平移"的简化版聚焦成像是不是一回事？** 数据采集方式不一样——教科书通常默认发射和接收孔径都以扫描线为中心一起平移（这其实更接近下文的 `WideBeam`），而 128RyLns 受 MUX 硬件所限，把"读取128阵元数据"这个动作拆成了两次固定区块，跟"如何用这些数据做聚焦重建"分开成了两层。但两者在**最终有效接收孔径**上是等价的——因为重建阶段本来就是"挑F值范围内的近处阵元求和、远处的哪怕在缓冲区里也不参与运算"，这跟"当初就只接收了那一小撮阵元"在计算结果上没有区别。**真正的差异只在于"每次发射能攒出多大的原始channel数据池供你挑"**：`WideBeam` 的池子上限是64（发射时连接了多少就只有多少），`128RyLns` 靠多打一次把池子扩大到128。

这值不值得？用 L11-5gH 的真实数字算一下 F 值（阵元间距0.3mm，128阵元宽≈38.1mm，128RyLns 默认最大成像深度≈38.9mm）：

```
64阵元孔径：  F# = 38.9 / 18.9 ≈ 2.06   （偏窄，横向分辨率打折）
128阵元孔径： F# = 38.9 / 38.1 ≈ 1.02   （接近F/1，典型的"好"值）
```

在这个深度上，64阵元明显不够，这正是 128RyLns 愿意为"能用满128阵元做动态接收孔径"多付出一倍发射代价的真实原因——不是没事拆着玩。

### 策略 C2：小孔径滑窗 + 窗口内接收（`WideBeam` 系列）

发射端跟 128RyLns 几乎一样（小孔径滑动窗口），但**接收端直接复用当前连接窗口**，不再多打一次去覆盖另外64个阵元：

```matlab
Receive(...).Apod = Papod(j,:);   % 直接用当前64阵元连接窗口接收，不额外分两半
```

所以每条射线只需要发射+接收一次，不存在"同一线打两次"的惩罚。代价是接收孔径被硬性限制在发射时连接的那64阵元以内，不能像 128RyLns 那样动态调用全部128个阵元——这是"帧率 vs 动态接收孔径宽度"的取舍，两种脚本代表两种设计选择。

**"Wide" 到底体现在哪？** 发射孔径（`numTx` 默认32）本身并不大，"宽"体现在两处，都不是"发射用了更多阵元"：

1. **发射焦点被故意设得极远**，波束在成像范围内基本不收拢——这才是"宽波束"名字的真正来源：

   ```matlab
   % 128RyLns（真聚焦）：TX.focus = 100 波长，落在成像深度内
   % WideBeam （宽波束）：TX.focus = 3*P.endDepth = 576 波长，远超成像深度，几乎不收拢
   ```

2. **每次发射照亮的横向范围宽**，所以一次发射就能重建一条"宽"的图像区域，不需要像 128RyLns 那样每个像素列都单独打一次。`PData.Region` 宽度对比：128RyLns 每次发射只对应 1 个阵元间距宽的窄条（标准逐线扫描），WideBeam 每次发射对应 `(numTx-12)=20` 个阵元间距宽的宽条（约6mm）。默认只用 48 次发射（而非 128RyLns 的128次）就能靠宽条重叠覆盖整个视野，这是宽波束帧率更高的根本原因。

   接收孔径（64，MUX连接窗口）确实比发射时实际驱动的孔径（约32，做过 kaiser 窗apod）更宽——用较窄孔径产生弱聚焦的宽波束发射，接收则用更宽的孔径做动态聚焦，弥补发射侧弱聚焦损失的部分横向分辨率。

把三种发射方式按"发射焦点距离"排成一条连续谱，`WideBeam` 正好卡在中间：

```
聚焦成像(128RyLns)          宽波束(WideBeam)              平面波(Flash)
focus=100波长(成像范围内)    focus=576波长(范围外3倍,弱聚焦)   focus=0/无穷远(完全不收拢)
128次发射覆盖128线           48次发射覆盖全阵列                3次(×na)覆盖全阵列
单点分辨率最优               分辨率/帧率折中                  帧率最高，质量依赖后续复合
```

这也是为什么临床上"宽波束"常被当作性价比最高的日常扫查默认模式。

## 3. 64 通道的限制到底卡在发射还是接收？

**接收才是硬限制，永远绕不开；发射端可以靠"配对绑定"或"缩小孔径+滑窗"规避 64 通道限制，接收端做不到任何一种规避**——接收必须让每个阵元的回波独立数字化才能用于波束合成，"把两个阵元接到一个通道"在接收端等于直接把两路信号相加、销毁空间信息，物理上不可逆。

由此可以直接回答"64通道设备 vs 原生256通道设备到底差在哪"：

| 场景 | 64通道MUX需要的采集次数 | 256通道系统需要的次数 | 差距 |
|---|---|---|---|
| AcquireRF（全阵元平面波） | 1发射+2接收 | 1发射+1接收 | 差距较小 |
| Flash/FlashAngles（3窗口平面波） | 3(×na) | 理论1(×na)即可 | **明显，拖慢到1/3** |
| 128RyLns（接收要覆盖全128阵元） | 每线2次 | 每线1次 | **2倍** |
| WideBeam（接收主动限制在64内） | 每线1次 | 每线1次 | **基本等价，MUX隐身** |

也就是说，"发射孔径 < 64" 只保证发射端不用绕路；**接收孔径是否 > 64（是否要用超过64个物理阵元的回波数据做重建）才是决定要不要多付出采集次数代价的真正开关**。128RyLns 为图像质量选择了"大接收孔径"，所以要付出双倍发射的代价；WideBeam 为帧率选择了"接收孔径=发射孔径"，所以在64通道系统上几乎不吃亏。

## 4. 接收端是怎么"聚焦"的：纯软件动态波束合成

接收聚焦完全是数字信号处理层面的事，没有任何物理/模拟层面的"接收聚焦"：

1. 每个开启的阵元各自独立采集、独立数字化——只是"忠实记录每个阵元听到了什么"。
2. 波束合成阶段，对图像里每一个像素点（每个深度），系统反算"这个点的回波传到每个阵元需要多久"，给每路信号施加对应延迟补偿使其对齐，再加权求和（经典 Delay-and-Sum）。
3. 这个延迟补偿是**逐深度重新计算**的，不是发射时预设好就固定不变——这是接收端理论上能在每个深度都完美聚焦的原因。这跟发射端完全不同：发射延迟波形一旦打出去就无法再改，一次发射只能精确聚焦在一个深度，越远离该深度聚焦质量越差。**发射聚焦是静态的、只能选一个焦点；接收聚焦是动态的、可以对每个深度重新聚焦**——这条不对称性是超声波束合成的基本常识。
4. 接收端通常还会做**动态孔径**（深度越大参与波束合成的阵元范围自动放宽以维持F值）和**切趾加权**（apodization，压制旁瓣）——同样是纯数字运算。

## 5. 帧率是怎么算出来的

### 5.1 `timeToNextAcq`：不是"事件间等待"，而是 PRI 设定器

官方《Sequence Programming Manual》对 `'timeToNextAcq'` 的定义（第58页）：该指令在带发射的 Event 里放置时，从**这次发射开始的那一刻**启动一个计时器，下一次发射必须等计时器到期才能开始；如果因为别的原因（处理没跟上、TPC切换未完成）导致延迟，系统不报错，只在命令行给一条警告，然后继续——**这是一个下限，不是精确定时器，实际间隔只会更长不会更短**。本质上就是设置脉冲重复间隔 PRI（=1/PRF），这也是为什么对 PRF 抖动极其敏感的多普勒采集必须靠它来控制节奏。

### 5.2 TTNA 公式与 384 波长天花板

```matlab
TTNA = round(2*384*(1/Trans.frequency)) + 20;
% 384   = 假设的最大成像深度（波长数），是跨探头复用的保守安全余量，不是精确计算值
% 2*384 = 声波往返总波长数
% +20   = HVMux开关建立时间 + 系统开销（微秒）
```

L11-5gH（7.6MHz）：`TTNA = round(2×384/7.6)+20 ≈ 121us`。**这个384波长上限不会随UI深度滑块自动变化**——RangeChangeCallback 只更新 PData/TGC，不重算 TTNA，所以默认脚本里成像深度和序列等待时间是脱钩的。同一份公式在频率完全不同的 P4-2gH（3.15MHz）脚本里原样出现，证明"384"是通用安全系数而非探头专属最优值。

波长换算物理深度：`λ(mm) = 声速(m/s) / 频率(MHz) / 1000`。L11-5gH：`λ≈0.2026mm`，384波长≈77.8mm单程——可以反推验证往返时间 `2×77.8/1.54≈101.05us`，正好对应公式里 `2×384/7.6` 那一项（声速项在换算中被约掉了）。128RyLns/Flash 默认成像深度192波长≈38.9mm，远小于77.8mm天花板，留了约2倍余量。

### 5.3 一帧多少次采集，帧间隔怎么"凑"出来

```
每帧采集次数 = 合成孔径切片数(2或3) × 转向角度数 na
```

`na`（转向角）不是平面波专属——凡"多角度复合"技术都用得到：`FlashAngles`/`FlashAnglesDoppler`（平面波相干复合）和 `WideBeamSC`（空间复合，3角度）都会设置 `TX.Steer`；纯逐线聚焦（`128RyLns`）和不带复合的宽波束（`WideBeam`/`WideBeamDoppler`/`WideBeamHI`）因为每个位置只需一次采集，`TX.Steer` 恒为 `[0,0]`，没有这个参数。

以 `FlashAngles` 为例，脚本把帧周期硬编码"凑"到固定 20ms：

```matlab
SeqControl(3).argument = 20000 - (3*na - 1)*TTNA;   % 补齐到 20000us
```

无论 `na` 取多少，只要 `(3×na−1)×TTNA < 20000`，脚本都会把总周期填到固定20ms，目标帧率恒为50Hz——这是作者刻意选择的序列速率上限，不是硬件极限。

另外，`frameRateFactor` **不是**帧率参数：它控制"每采集完多少帧才把控制权交还MATLAB"，影响的是软件层面多久检查一次状态，不改变硬件真实采集触发周期。

### 5.4 实际显示帧率 ≠ 序列触发帧率

处理（`Recon`/`Process`/显示刷新）是**异步**于采集执行的。真正决定屏幕帧率的隐藏变量是：**PC/GPU处理这一帧所需的时间，如果超过序列触发周期，处理端就会成为瓶颈**。无复合的 `Flash`（每帧3次采集）处理量小，通常能跑满50fps目标；`FlashAngles`（na=7，21次子孔径重建叠加）处理耗时明显增加，实际显示帧率可能明显低于50fps目标。

## 6. 理论帧率天花板，以及为什么脚本只给了 50fps

按 L11-5gH 的 TTNA≈121us、Flash 每帧3次采集算理论下限：

```
理论最短帧周期 ≈ 3×TTNA = 363us   →   理论最高帧率 ≈ 2755 fps
```

脚本实际把帧周期拉长到20000us，只用了理论能力的约1.8%。这不是失误，而是官方手册在"7.2.3 Asynchronous Acquisition and Processing"一节明确讨论过的设计取舍，原文举的例子正是**剪切波弹性成像**（可以做到每秒数千帧，之后离线批处理）。这批"日常B超"示例脚本没有去够那个极限，主要三个原因：

1. **给异步处理留安全余量**——`timeToNextAcq` 设得太接近软件处理耗时，一旦系统偶尔卡顿就会持续触发"missed"警告；50fps对普通B超实时重建+显示通常游刃有余。
2. **临床显示不需要更高帧率**——人眼可感知流畅视频约30-60fps已够，组织运动比这慢得多；真正需要千帧级的场景（剪切波、超快血流）需要专门设计流水线，不是调小一个常数就能达成的。
3. **高压发射电路和换能器本身有速率上限**——长时间高PRF发射可能导致电源电压跌落（droop）、驱动电路发热超限，这是纯软件层面调不掉的硬件约束。

**推论：相干平面波复合（Coherent Plane-Wave Compounding, Montaldo 2009）用几十帧换上百帧最终帧率是真实可行的**，去掉人为20ms填充、只按理论最小值算：

```
na张平面波复合一帧耗时 ≈ na × 363us
```

| na | 耗时 | 理论帧率 |
|---|---|---|
| 10 | 3.6ms | ≈275 fps |
| 20 | 7.3ms | ≈138 fps |
| 27 | 9.8ms | ≈102 fps（"上百"分界点） |
| 50 | 18.2ms | ≈55 fps |

前提是：① 必须动手把 `SeqControl` 里"锁定20ms"的填充逻辑改掉，否则默认脚本无论 `na` 多大帧率都纹丝不动；② 需要能跟上速度的（通常GPU加速的）IQ域相干求和流水线；③ 仍受高压电源/散热占空比限制；④ 数据搬运带宽要跟上。

## 7. 实战选型：帧率 vs 图像质量

| 脚本 | 每帧采集次数 | 目标周期 | 目标帧率 | 说明 |
|---|---|---|---|---|
| `AcquireRF` | 2 | 10ms | 100fps | 无实时B超显示流水线，纯RF采集模板 |
| `Flash` | 3 | 20ms | 50fps | **追求帧率的首选**——处理量最轻，实测最接近目标值 |
| `WideBeam` | 48 | 20ms（靠补齐） | 50fps（虚高） | 采集/重建负担远高于Flash，实际达不到 |
| `FlashAngles`(na=7) | 21 | 20ms（靠补齐） | 50fps（虚高） | 重建负担是Flash的7倍 |
| `128RyLns` | 256 | 50ms | 20fps | 图像质量优先，帧率天生垫底 |
| `WideBeamHI`/`SC`/`Doppler` | WideBeam基础上再×2~3 | ~20ms | 虚高 | 质量优先，不适合追求帧率 |

`High_Image_Quality` 三脚本各自的质量提升机制：**`WideBeamHI`**（脉冲反转谐波成像）——谐波信号集中在声束中心、旁瓣贡献少，且不受浅层线性混响干扰，近场杂波显著减少、对比度提升；**`WideBeamSC`**（3角度空间复合）——不同角度的斑点噪声弱相关，平均后显著抑制，同时能"绕开"单角度镜面反射造成的信号缺失；**`WideBeamDoppler`**——不提升B超本身画质，而是增加血流功能信息。三者共同代价都是降帧率（HI多打一次、SC多打3倍、Doppler插入独立ensemble）。

`WideBeamHI` 的 "HI" = Harmonic Imaging（谐波成像），实现方式是脉冲反转（正相+反相各打一次，线性基波分量抵消、非线性二次谐波保留叠加）——跟临床设备的组织谐波成像（THI）是同一原理、同一套主流商业实现方案。

`WideBeamSC` 的复合角度数（3个）技术上可以增加，但代码把"3"硬编码在结构里（三段独立for循环+三个独立RcvBuffer），不像 `FlashAngles` 的 `na` 是可直接调的循环变量，需要改写为通用循环结构才能扩展。

## 8. Doppler 双工模式：WideBeamDoppler vs FlashAnglesDoppler

两个脚本的B超部分沿用各自策略（WideBeam / FlashAngles），但 **Doppler 部分的波束设计思路也完全不同，且跟各自B超部分是同构的**：

| | `WideBeamDoppler` | `FlashAnglesDoppler` |
|---|---|---|
| Doppler波束数 | 4条独立窄波束（各40阵元孔径），分布在阵列不同位置 | 1条，覆盖固定64阵元窗口 |
| 覆盖方式 | 多条窄波束拼出整个ROI（同构于WideBeam"多次宽条覆盖"） | 单次"类平面波"宽发射照亮整个ROI，靠接收动态孔径重建多点（同构于Flash"少数宽发射覆盖全视野"） |
| ensemble采集方式 | **交叉扫描**：ensemble长度16，4条波束依次轮流打，每条波束实际PRF=设定值，触发间隔=1/(PRF×4) | **不交叉**：单一波束连续打14次 |
| 每帧Doppler采集次数 | 4×16=64次 | 14次 |
| 界面ROI框显示 | 有专门的外部函数画框 | 无 |

多波束交叉扫描是官方手册里"低PRF时交叉扫描多条线以提升有效帧率"技巧的教科书应用。

**Verasonics 的 Doppler 处理流水线**（两个脚本共用同一套，源自《Sequence Programming Manual》第22章）：

1. **物理原理**：`Δf = (v/c)·2f₀·cos(θ)`——血流散射信号比组织回波弱40-60dB，宽带发射脉冲里测不出直接频移，所以测的是**连续多次发射之间回波的相位变化率**。
2. **采集（ensemble）**：同一波束方向重复发射/接收N次（典型8-20次），PRF需略大于预期最大多普勒频移的2倍防止混叠；PRF较低时可以交叉扫描多条线提升有效帧率。
3. **波束合成**：ensemble里每次回波独立做延迟叠加，得到一份IQ数据，存进InterBuffer的一"页"（多页缓冲区，一页对应ensemble里一次采集）。
4. **Doppler处理**（`Process.classname='Doppler'`）：
   - **壁滤波**（`wallFilter='regression'`，二次回归高通）：沿"慢时间轴"（ensemble方向）滤掉组织运动等强而慢的杂波，只留血流引起的快速相位变化；处理从第3页开始（跳过前两页的滤波器建立瞬态）。
   - **自相关估计**：官方手册明确这是"Doppler Process Object 内部实现的自相关处理"（经典Kasai型算法）——计算相邻ensemble采样的平均相位差，换算成平均多普勒频移/速度（`computeCFIFreqEst`，输出-128~127映射到-PRF/2~+PRF/2）；或只统计回波功率不管方向（`computeCFIPowerEst`，功率多普勒，输出0~255）。
   - **阈值筛选**：`pwrThreshold`滤掉太弱的估计（当噪声），`maxPower`滤掉过强的估计（当组织运动剔除）。
   - **postFilter**（可选）：对估计结果做空间平滑。
5. **显示**：写入ImageBuffer，用`classname='Image'`叠加到B超灰阶图，显示彩色血流图（速度模式`signedColor`红蓝双向，功率模式`unsignedColor`单色能量）。
